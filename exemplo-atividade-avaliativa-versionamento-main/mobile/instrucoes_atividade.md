# Atividade Prática Integradora: Portfólio de Projetos (Hub Técnico)

## 📌 Descrição Geral
Nesta atividade avaliativa integradora de **Versionamento de Código**, você criará um **Portfólio de Projetos (Hub Técnico)** no GitHub. O objetivo é duplo:
1.  **Vitrine Profissional:** Organizar de forma impecável e documentar as principais atividades práticas e projetos que você desenvolveu nas disciplinas técnicas deste bimestre.
2.  **Gestão Profissional de Código:** Demonstrar domínio prático das ferramentas de versionamento fundamentais para o mercado de trabalho, focando estritamente em **Conventional Commits** para a padronização do histórico de commits.

Você deve incluir no portfólio pelo menos uma atividade prática relevante de cada uma das seguintes disciplinas do curso:
*   **Mobile**
*   **Front-end**
*   **Back-end**
*   **Banco de Dados**
*   **Inteligência Artificial (IA)**
*   **Versionamento** (que é o próprio projeto deste portfólio)

## 🎯 Requisitos da Atividade

### 1. Estrutura do Repositório
O repositório deve ter a seguinte estrutura física de pastas:
```
portfolio-tecnico/
├── README.md                      # README principal do portfólio (vitrine técnica)
├── mobile/
│   ├── README.md                  # Explicação e código/arquivos da atividade de Mobile
│   └── [arquivos da atividade]
├── front-end/
│   ├── README.md                  # Explicação e código/arquivos da atividade de Front-end
│   └── [arquivos da atividade]
├── back-end/
│   ├── README.md                  # Explicação e código/arquivos da atividade de Back-end
│   └── [arquivos da atividade]
├── banco-de-dados/
│   ├── README.md                  # Explicação e scripts/arquivos de Banco de Dados
│   └── [arquivos da atividade]
├── ia/
│   ├── README.md                  # Explicação e arquivos/scripts de IA
│   └── [arquivos da atividade]
└── versionamento/
    └── README.md                  # Documentação das práticas de versionamento aplicadas
```

### 2. Padrão de Commits (Conventional Commits)
Seu repositório **não** pode conter commits genéricos (como `"ajustes"`, `"commit final"`, `"teste"`). Todo o histórico de commits deve ser profissional e seguir as especificações dos Commits Convencionais:
*   `feat`: Adição de um novo projeto, arquivo ou funcionalidade (ex: `feat(mobile): add weather app files`).
*   `fix`: Correção de bugs no código ou caminhos de arquivos (ex: `fix(backend): correct database port connection`).
*   `docs`: Alteração em documentações e READMEs (ex: `docs(readme): explain backend learning objectives`).
*   `style`: Alterações estéticas ou de formatação que não mudam a lógica (ex: `style(frontend): format index.css margins`).
*   `refactor`: Modificação de código para melhoria de legibilidade/estrutura (ex: `refactor(logica): modularize menu procedures`).
*   `chore`: Atualizações de configuração ou tarefas de rotina (ex: `chore: setup package.json properties`).

**Atenção:** Realize commits incrementais. O histórico deve refletir a evolução do portfólio passo a passo (mínimo de 6 commits estruturados ao longo do projeto).

## 📝 Documentação Obrigatória (O README Perfeito)
Cada subpasta de disciplina deve conter um arquivo `README.md` preenchido por você. Não copie definições de internet. A documentação deve responder:

1.  **O que aprendi:** Explicação teórica simples e em suas palavras sobre o principal conceito daquela matéria neste bimestre.
2.  **Para que serve (Por que aprendi):** Aplicação prática desse conceito no dia a dia de um desenvolvedor no mercado de trabalho.
3.  **Como funciona/Demonstração:** Instruções rápidas de como rodar o script/app ou um trecho de código comentado fundamental.

## 📊 Critérios de Avaliação (Rubrica de Notas)

| Critério | Descrição | Peso |
| :--- | :--- | :---: |
| **Padrão de Commits (Conventional Commits)** | Todos os commits seguem rigorosamente a convenção de *Conventional Commits* com escopos corretos, sem mensagens informais ou genéricas, mostrando a evolução incremental. | **35%** |
| **Qualidade da Documentação** | READMEs detalhados com as explicações reflexivas (o que aprendeu e para que serve) de cada uma das 6 disciplinas. | **45%** |
| **Organização do Repositório** | Estrutura correta de pastas, links internos funcionando no README principal e ausência de arquivos lixo ou temporários. | **20%** |

## 🚀 Como Começar
1. Crie um repositório **público** no GitHub.

3. Transfira a estrutura de pastas e arquivos para o seu repositório local.
4. Comece adicionando seus projetos e fazendo commits semânticos!
